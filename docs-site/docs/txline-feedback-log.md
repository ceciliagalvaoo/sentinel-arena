# TxLINE API Feedback Log

A required submission deliverable, kept as a running log since day one of development, each entry dated, describing what was attempted, and what turned out to be a genuine surprise (good or bad). Nothing here is retroactively cleaned up; it's the actual order things were discovered in.

**2026-07-07**, Project start. No real API calls made yet; monorepo scaffolding finished first so integration work could start directly in `txline-client` with no manual steps left pending mid-build.

**2026-07-08**, The full `auth → subscribe → activate` flow ran end-to-end on devnet for both wallets without friction: the reference docs matched real behavior exactly (activation message format, token response shape, etc.). The one detail the docs don't make obvious, clarified only by the official example repo (`tx-on-chain/examples/devnet/common/users.ts`), is that `subscribe()` requires the caller's Token-2022 Associated Token Account for the TxL mint to **already exist** before the call, even on the free tier where nothing is actually charged.

**2026-07-09**, An important finding for the replay-mode architecture: `GET /api/fixtures/snapshot` only returns fixtures from `startEpochDay` forward (default: today), so a bare call showed nothing from the World Cup (already past the phase mapped at the time). Passing the correct `startEpochDay`, or querying `odds/snapshot/{fixtureId}` / `scores/snapshot/{fixtureId}` directly with a known `fixtureId` from the schedule, surfaces the real, complete World Cup data on devnet. Tested against fixture 18209181 (France vs. Morocco, quarterfinal): detailed score history (goals, cards, corners, per-minute free kicks) and odds via `asOf` throughout the match. This reprioritized `recorder-service`: instead of only recording live events (risky, depends on catching a match in progress), a **historical REST backfill** of any already-finished fixture became possible at any time, substantially reducing the hackathon's calendar risk.

**2026-07-10**, `GET /api/scores/historical/{fixtureId}` responds with `Content-Type: text/event-stream` and an SSE-formatted body (`data: {...}\nid: N\n\n` per record), unlike every other snapshot/updates endpoint, which returns plain `application/json`. Not flagged in the OpenAPI reference, the table treats every endpoint uniformly. Only noticed after inspecting the raw response body once `response.json()` failed outright.

**2026-07-10**, A full backfill of a real World Cup fixture worked end-to-end via pure REST (no live SSE needed): fixture 18209181 (France vs. Morocco) yielded 1,116 score events and 34,145 odds events, much higher volume than expected for a single match (StablePrice updates many houses/markets in parallel). Confirmed `odds/updates/{epochDay}/{hourOfDay}/{interval}` correctly filters by `fixtureId` (751 records in one sampled 5-minute bucket, all the same fixture). An easy mistake avoided: anchoring the odds backfill window on the earliest `Ts` seen in `scores/historical` overshoots, since score coverage can start days before kickoff (`coverage_update`) with no odds ticking yet, anchored on the score record's own `StartTime` field instead.

**2026-07-10**, `GET /api/odds/stream` connects and stays stable with automatic JWT renewal wired in (the `eventsource` package v4 with a custom `fetch`, matching the official example repo's pattern), no errors across a 20-second connection even with no live match in progress (silence/heartbeat only, as the docs say to expect).

**2026-07-11**, Not about the TxLINE API itself, but about Solana's public devnet RPC (`api.devnet.solana.com`): it rate-limits aggressively (`429`) under a burst of transactions, discovered running the agent loop against an accelerated replay, which naturally generates several commits in quick succession. Fixed with a serialized transaction queue with minimum spacing between publishes; for real operation, a dedicated RPC provider (Helius, QuickNode, etc.) is worth it over the public endpoint, even on devnet.

**2026-07-11**, A real bug in the agent loop's own design: since `MarketDataSource.onOdds`/`onScore` are fire-and-forget by design (the source never awaits a handler), a `game_finalised` settlement could run **before** every earlier commit for that fixture had actually persisted, and the pending-signals lookup simply wouldn't find those still "in flight", silently losing their reveal/grade. Fixed by enqueuing the entire settlement as one unit on the same serialized queue used for commits, guaranteeing real FIFO order. End-to-end validation against a real fixture: 286 signals → 285 commits → 143 reveals (first pass) → 143/143 reveals with hash verified matching the commit, proving the full cycle's cryptographic integrity in real production (devnet), not just in theory.

**2026-07-11**, The reference doc's `validateStatV2` payload example is illustrative only, real field names only match exactly what's in the vendored IDL (`idl/devnet/txoracle.json`). A wrong field name here doesn't produce a TypeScript compile error (the client is typed against the generic `anchor.Idl`, not a literal generated type), it only fails at runtime during on-chain deserialization. Checked every struct (`StatValidationInput`, `ScoresBatchSummary`, `StatLeaf`, `NDimensionalStrategy`, `StatPredicate`, `TraderPredicate`) field-by-field against the IDL before writing the client. Tested the validation against the real final score of fixture 18209181: requesting confirmation of the real result (France won) returns `true`; requesting confirmation of the wrong result returns `false`, confirming both the Merkle proof and the goal-margin predicate (`subtract` + `comparison`) are correct.

**2026-07-12**, `validateOdds` has a stricter timestamp requirement than `validateStatV2`, discovered only by testing against devnet directly (not documented anywhere): for scores, the `ts` argument is the batch summary's `minTimestamp`; for odds, it must be **exactly** the individual tick's own `ts`, using the batch minimum fails on-chain with `AnchorError: TimestampMismatch`. Tested end-to-end once corrected: a genuine proof for a real odds tick returns `true`; the same proof with tampered prices (same Merkle tree, forged leaf) is rejected on-chain with `InvalidSubTreeProof`, the same rigor already proven for the score-validation path.
